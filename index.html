📚 Definitions & Key Terminology
Architecture
Humbl AgentGlasses hardware orchestration layer. Deterministic firmware handling sensors, mic, camera, LEDs, haptics. No AI runs on glasses. Humbl ConnectCompanion mobile app (Flutter, iOS + Android). Hosts the full AI pipeline: STT, SLM, cloud LLM, memory, tools, TTS. StateGraphFully deterministic orchestration engine. Manages Stage 3→4→5 routing, loop-backs, interrupts, and BG agent dispatch. Not an LLM. Pipeline RunA single end-to-end execution from trigger to response. Identified by unique run_id. Can be paused, resumed, or promoted to BG agent. Stage 3 / 4 / 5The 3-stage pipeline: Think (intelligence + routing) → Do (resource acquisition + tool execution) → Deliver (output + persist + memory write).
AI & Machine Learning
SLMSmall Language Model. On-device LLM (Qwen3-0.6B, ~400MB Q4 quantized) for fast intent classification, parameter extraction, and light reasoning. Runs via llama.cpp FFI. LLMLarge Language Model. Cloud-hosted models (GPT-4o, Claude, Gemini, etc.) accessed via LLM Gateway for complex reasoning, multi-step planning, and open-ended conversation. KV CacheKey-Value cache from SLM inference. Preserves conversation state between turns (~120MB). Handed off to cloud LLM context when escalating. Evictable under memory pressure. GGUF / Q4_K_MModel quantization format. GGUF is the container; Q4_K_M is 4-bit mixed quantization balancing quality and size. Used by llama.cpp. EmbeddingDense vector representation of text (384-dim via all-MiniLM-L6-v2, ~23MB ONNX int8). Used for semantic search in T3 memory tier and RAG retrieval. RAGRetrieval-Augmented Generation. Combines vector search results with LLM prompts to ground responses in user’s personal data and past interactions. VADVoice Activity Detection. Detects speech start/end (Silero neural ~2MB, WebRTC rule-based fallback). Gates STT to avoid processing silence. STTSpeech-to-Text. Converts audio to text on-device (Whisper.cpp, ~75MB GGML). Runs after VAD detects speech end. TTSText-to-Speech. Converts AI response to audio. 4-tier: earcon (pre-recorded) → template+slot → Piper neural (~50MB) → System TTS fallback.
Memory & Storage
Hive (Performance Cache)NOT a memory tier. Performance cache layer sitting IN FRONT of pipeline. 6 boxes: llm, rag, vision, tts, routing, session. TTL-based eviction. ~5–100MB (tier-dependent). Caches LLM responses, RAG results, vision outputs, TTS audio for sub-millisecond repeat access. See Hive deep dive. T2 (KV Store)Structured key-value tier. SQLite-backed patterns, preferences, and user facts. “Mom = mobile”, “preferred_volume = 70%”. Survives app restart. T3 (Vector)Semantic search tier. sqlite-vec with 384-dim embeddings. Natural language queries retrieve relevant past interactions, documents, and context. T4 (Log)Full trace tier. Append-only SQLite log of every interaction: full transcript, tool calls, timing, tokens used. Backs cloud sync to Supabase. sqlite-vecSQLite extension for vector similarity search. Enables on-device semantic retrieval without a separate vector database. HiveLightweight pure-Dart NoSQL key-value store. Binary box files for fast local caching. Used as T1 cache tier (6 boxes: llm, rag, vision, tts, routing, session). Data SavingPersisting to local storage (SQLite T2–T4 + Hive on disk). Survives app restart. Prerequisite for backup. Data BackupSyncing local storage to cloud (Supabase). Cross-device recovery + disaster recovery. Cannot back up without first saving. Importance ScoringEvery T2/T3 memory entry scored 0.0–1.0 at write time. Eviction targets lowest-importance entries first under storage pressure. No time-based TTL windows. High-importance memories survive indefinitely. Tier-dependent protected pool size (50–2000 entries). Memory ConsolidationHybrid pipeline (embedding clustering + SLM classification + template compaction) that reviews aged memories, deduplicates, prunes, and compacts for long-term archive. Standard+ only. Triggered by storage pressure. Context RolloverAutomatic agent session continuation when context window nears capacity (~80%). Agent Manager monitors, agent compacts its own context, new session spawns with compact summary. Transparent to user. BYOKBring Your Own Key. All users can provide own LLM API keys (OpenAI, Anthropic, Google). Free tier’s only cloud LLM path. Keys never leave device (AES-256 encrypted in Hive). Not synced to cloud.
Connectivity & Hardware
BLEBluetooth Low Energy. Primary wireless link between glasses and phone. Used for control plane events, audio streaming (LC3/ISO), and sensor data. A2DPAdvanced Audio Distribution Profile. Bluetooth profile for high-quality one-way audio streaming. Used for TTS output from phone to glasses speaker. USBHighest priority wired connection. Carries all communications (control, audio, data, camera) when physically connected. Preferred over BLE. FCM / APNsFirebase Cloud Messaging / Apple Push Notification service. Push channels for waking the app when BG agents complete or cloud events arrive. ProtobufProtocol Buffers (nanopb). Compact binary serialization for Agent↔Connect messages. Versioned envelope with integrity checks.
Operations & Observability
SLOService Level Objective. Target reliability/latency metric (e.g., 99.5% availability, <200ms latency). Measured over rolling window. RPORecovery Point Objective. Maximum tolerable data loss measured in time. RPO 15min = at most 15 minutes of data lost after failure. RTORecovery Time Objective. Maximum tolerable downtime. RTO 60min = system must be restored within 60 minutes of failure detection. CAPEXCapital Expenditure. One-time upfront costs: hardware prototypes, wake word licensing, BT qualification, test devices. OPEXOperating Expenditure. Recurring monthly costs: cloud LLM API tokens, server hosting, storage, egress, observability, app store fees. trace_idUnique identifier carried through every event in a pipeline run. Enables end-to-end correlation across Agent → Connect → Backend for debugging. SupabaseSelf-hosted open-source backend (PostgreSQL + Auth + Realtime + Storage + Edge Functions). Replaces Firebase with full data ownership.

🚀 BG Agent Dispatch (finite async only)
Two sources:
Dispatched: SLM or cloud includes bg_dispatch → Stage 4 starts new finite BG agent.
Promoted: User interrupts current run and chooses “background it” → current run’s StateGraph state serialized to Hive/SQLite → continues as BG agent.
Both managed by Agent Manager. Result returns to Stage 3 when complete.
⚡
Interrupt Exit Point — if user/event trigger arrives during Stage 3 reasoning: terminate, pause + new, continue, or background current run (promote to BG agent). System interrupts (OS) yield immediately.
🔐 Tier-Aware Routing — Cloud Gate Check
Before any cloud route (Path 3 Cloud Loop, Path 4 Cloud Always), StateGraph checks tier entitlement via the quota chain:
// StateGraph cloud gate — runs before LLM Gateway dispatch
if toggle == privacy || toggle == cloud_less → force SLM path
if tier.cloud_quota > 0 → route to cloud (deduct quota)
if tier.topup_balance > 0 → route to cloud (deduct top-up)
if tier.byok_key != null → route to cloud via user’s key
else → SLM generates gating response (user can follow up)
Gating response (SLM): When cloud is unavailable (quota exhausted, no BYOK, toggle active), SLM generates a natural language explanation: “I can’t use cloud AI right now — your daily quota is used up. Want me to handle this locally, or would you like to top up?” User can follow up conversationally.
Privacy / Cloud-less override: When either toggle is active, cloud routes are blocked at StateGraph level before LLM Gateway is even called. No DNS resolution, no network activity. SLM handles all requests. Privacy mode additionally limits context to RAM-only T1 cache (no T2/T3 persisted memory retrieval).
⚡ Stage 3 is the brain — all intelligence (SLM/Cloud), routing, and cloud internal reasoning. No execution, no TTS, no persistence. Routes to Stage 4 (do) or Stage 5 (speak). Tier gating checked before every cloud dispatch. Free: BYOK only (zero Humbl quota — no key configured → SLM fallback). Privacy mode: context limited to RAM-only T1 Hive cache (T2/T3 not persisted). Context rollover may trigger during long SLM/cloud loops.

Agent Manager
Manages all agent lifecycles — main pipeline runs (interrupt, pause, resume) and background agents (dispatched + promoted). Enforces concurrency limits (default 4 local, cloud BG tier-gated: Free 0 / Standard 2 / Plus 4 / Ultimate ∞). Monitors agent context window usage and triggers context rollover at ~80% capacity. 🔔 Agent Manager — StateGraph Interaction Points
Dispatch — Stage 3 outputs bg_dispatch → StateGraph calls Agent Mgr → register + start finite async agent
Promote — User interrupts + backgrounds run → StateGraph serializes state → Agent Mgr registers as promoted agent → continues execution
Complete — BG agent finishes → Agent Mgr delivers result to Stage 3 input queue → triggers new pipeline run or queues for next iteration
Lifecycle — Runs independently of pipeline. Monitors device state continuously. Can pause/throttle/terminate agents without StateGraph involvement.

🔐 Tier-Aware Execution
Cloud BG Agent Cap: Agent Manager checks active_cloud_agents < tier.max_cloud_bg before dispatch. Free: 0, Standard: 2, Plus: 4, Ultimate: ∞ (fair-use). If cap reached, SLM responds: “All your cloud agent slots are in use. Want me to handle this locally instead?”
Cloud-Dependent Tools: Tools requiring cloud LLM (e.g., complex summarization, translation beyond SLM capacity) follow the same quota chain as Stage 3 routing. Cloud-less / Privacy toggle: cloud tools unavailable, SLM notifies user of limitation and offers local alternatives.
🚀 Stage 4 is the hands — pure execution, no intelligence. Resource Manager provides context + HW access. Tool Execution runs 121 MCP-described tools (118 core + 3 subsystem). Agent Manager handles agent lifecycle (cloud BG cap: Free 0, Standard 2, Plus 4, Ultimate ∞) and monitors agent context window usage for rollover. Results loop to Stage 3 or forward to Stage 5.

🔐 Tier-Aware Persistence & Output
Privacy Mode ON: All disk writes skipped. T2/T3/T4 disabled. T1 Hive operates in RAM-only mode. TTS/display output still delivered normally. Journal + telemetry disabled. No Sentry. Nothing survives app close.
Free Tier: Data saving ✓ (full local T2/T3/T4, importance-scored eviction, hybrid consolidation). Cloud LLM via BYOK only (zero Humbl quota). Data backup ✗ (local persistence only). Protected pool: local only. Basic Sentry telemetry. No toggles.
Standard: Full data saving (persistent T1–T4). Data backup: Supabase best-effort OS-scheduled sync (1 GB). Hybrid memory consolidation. Full telemetry + trace_id correlation.
Plus / Ultimate: Everything in Standard + larger cloud quota (5 / 20 GB). Larger Hive cache + protected pool. Content versioned and synced across devices.
🔊 Stage 5 is multi-modal output — TTS (spoken), glasses display (HUD), companion app (text/UI), sound notifications (earcons), LED/light indicators, haptic feedback. Delivers persistent content, persists state + versioned content refs to memory (tier-gated: Privacy → RAM-only T1, Free/Standard/Plus/Ultimate → importance-scored eviction + hybrid consolidation. Free: local only, no cloud backup. Standard/Plus/Ultimate: best-effort OS-scheduled sync (1/5/20 GB per tier)). Memory consolidation runs as BG agent (all tiers) when storage pressure detected. Can also be invoked as tool via StateGraph. Decides if pipeline loops or completes.
Stage 6 — Background Maintenance (Maintain)
STAGE 6 — AUTONOMOUS + TOOL-INVOCABLE · POST-PIPELINE · ALL TIERS
Three independent subsystems that keep the device healthy. Each runs autonomously on its own schedule AND can be invoked as MCP tools via StateGraph (Stage 4) when user requests it. Post-pipeline — never blocks Stages 1-5. All tiers get the same local maintenance; only cloud backup requires Standard+.

Auto trigger: Best-effort OS-scheduled (iOS BGProcessingTask / Android WorkManager). Same frequency all paid tiers.
Tool trigger: User: “back up now”

🔌 Dual Invocation Model — Autonomous + Tool
Autonomous (default): Each subsystem monitors its own trigger conditions and runs independently. No pipeline involvement. Invisible to user. Yields to foreground work.
Tool-invoked (on demand): User asks via natural language → SLM/LLM emits MCP tool call (lora_train, memory_consolidate, cloud_backup_now) → Stage 4 dispatches → subsystem runs immediately with feedback.
📦 CONSOLIDATION: T2+T3 storage >80% → cluster → score → compact → prune → re-embed → storage reclaimed
🧠 LoRA: T4 log (append-only) → staging buffer ≥500 → device idle → QLoRA train → validate → adapter_vN.bin → merge @ cold start
☁️ BACKUP: T1-T4 + artifacts + adapter + profile → delta diff → Supabase upload → verify → available for restore/handoff
⚙️ Stage 6 is housekeeping — memory compaction, model personalization, and cloud sync. Runs autonomously in background (never blocks Stages 1-5) and can also be invoked as MCP tools. All three are independent subsystems that share nothing except device resources managed by Resource Manager. All tiers get local maintenance; cloud backup requires Standard+.

Voice Input System — VAD + STT
🎙️ HOW VOICE BECOMES TEXT — one of multiple input paths (see also: file/media/link payload attachment in Stage 1-2)
💡 Why
The glasses have limited or no screen for text input — voice is the primary glasses input interface. The voice input system must convert natural speech into clean text for the SLM to classify, while being battery-efficient (mic is not always-on) and fast enough for conversational latency (<500ms from end-of-speech to text ready). VAD prevents wasting compute on silence, and STT must handle noisy real-world environments (street, office, wind). Note: voice is optional — quick actions, file shares, and other event triggers can bypass voice entirely and send payloads directly to Stage 3.
⏱️ When
Mic Activates
Wake word detected (“Hey Humbl”) or AI button pressed on glasses. Mic is off by default — hardware-gated, not software. Battery-safe.
VAD Runs
Immediately after mic activates. Monitors audio stream for speech onset and silence. When user finishes speaking (silence threshold crossed), VAD signals end-of-utterance.
STT Runs
After VAD signals end-of-speech. Processes the captured audio buffer. Also runs during SLM follow-up loops (ask user → mic → VAD → STT) and cloud follow-up loops.
Bypassed
Text input from companion app (typed text) skips VAD + STT entirely → goes straight to SLM. Push notifications and scheduled triggers also bypass voice input.
⚙️ How
🔇 VAD — Voice Activity Detection
Model: Silero VAD (primary) / WebRTC VAD (fallback). Silero is a small neural network (~2MB), WebRTC is rule-based signal processing.
Latency: ~30ms per frame. Runs on CPU, no GPU needed.
Input: Raw PCM audio from glasses mic via BLE Audio (LC3) or USB.
Output: Binary signal — speech_active or silence_detected. Silence threshold: ~500-800ms of non-speech triggers end-of-utterance.
Noise handling: Silero trained on noisy datasets (SNR 0-20dB). Distinguishes speech from traffic, music, wind, keyboard.
BG Captions mode: VAD also gates the Live Captions background agent — only transcribes when speech is detected, saving STT compute.
🎙️ STT — Speech-to-Text
Model: Whisper.cpp (tiny/base model, ~75MB GGML). C++ inference via Dart FFI. Runs entirely on-device.
Latency: ~300ms for typical utterance (1-5 seconds of speech). Cold load ~500ms, warm ~300ms.
Input: Audio buffer from VAD-gated capture. 16kHz mono PCM.
Output: Transcribed text string. Passed to SLM for classification. { text: "set brightness to seventy percent" }
Languages: Multilingual Whisper supports 99+ languages. Model selection configurable per user language preference.
GPU/NPU: Uses NPU acceleration where available (Core ML on iOS, NNAPI on Android). Falls back to CPU if unavailable. Shares GPU budget with SLM via Resource Manager.
Streaming: Yes — incremental partials displayed on glasses HUD / companion app WHILE user speaks (visual feedback, like Google Assistant / Siri UX). Partials are display-only — they do NOT trigger pipeline. Pipeline triggers only after VAD end-of-speech → final transcript. Moonshine (MIT, streaming-optimized) used for incremental decoding with KV cache; Whisper.cpp for final accuracy pass. Live Captions BG agent uses same streaming path.
🔄 End-to-End Voice Flow
1. Wake word / button → glasses mic activates → audio streams via BLE to phone
2. VAD (Silero) monitors stream → detects speech onset → starts buffering audio
3. User speaks: “What’s on my schedule today?”
3a. Streaming STT (Moonshine) shows partials on HUD: “What’s on my...” → “What’s on my schedule...” (display-only, no pipeline trigger)
4. VAD detects ~600ms silence → signals end_of_utterance
5. Final STT (Whisper) receives audio buffer → transcribes in ~300ms → “What's on my schedule today?”
6. Text → Context Assembly (memory inject) → SLM processes
   // If SLM needs follow-up: TTS asks → mic reactivates → steps 2-6 repeat with KV warm
SLM — On-Device Intelligence (Qwen3-0.6B)
🧠 THE BRAIN ON YOUR PHONE — first line of intelligence before cloud
💡 Why
Most voice commands are simple — “call Mom”, “set brightness to 70%”, “start live captions”. These don’t need a cloud LLM. The SLM handles ~70-80% of all user requests entirely on-device: instant response, zero network dependency, no API cost, full privacy. Only complex reasoning (“analyze my schedule and suggest when to leave”) escalates to cloud. This keeps the system fast (sub-second for common tasks), cheap (no tokens burned for simple intents), and resilient (works offline).
⏱️ When
Every Request
SLM processes every single input from the pipeline. It’s always the first intelligence layer. Even requests that end up at cloud pass through SLM first for classification.
Handles Directly
Intent classification, parameter extraction, conversation loops for missing params, BG agent dispatch, routine workflows, deterministic multi-step commands.
Escalates to Cloud
Open-ended reasoning, multi-tool planning, creative generation, long-form content, ambiguous queries the SLM isn’t confident about (confidence < threshold).
Follow-up Turns
When SLM asks user for missing info, the response comes back to SLM with KV cache warm (~400ms). SLM maintains conversation state across turns without re-processing.
⚙️ How
🧠 Model Specs
Model: Qwen3-0.6B — 0.6 billion parameter generative language model. Small enough for phone, large enough for intent classification + structured output.
Quantization: GGUF Q4_K_M (~400MB on disk). 4-bit quantized for mobile inference. Minimal quality loss vs full precision for classification tasks.
Runtime: llama_cpp_dart (Dart FFI bindings to llama.cpp C++ engine). Runs on CPU + GPU/NPU where available.
Memory: ~400MB model weight + ~120MB KV cache at runtime. Total ~520MB. Fits comfortably on 8GB+ devices.
Latency: ~800ms cold start (first inference). ~400ms KV warm (follow-up turns reuse attention cache).
🔬 What SLM Outputs
Structured JSON: SLM is prompted to output machine-parseable JSON, not natural language. This makes routing deterministic — StateGraph reads the JSON, not interpreting prose.
Intent: One of ~100 classified intents (e.g. call_make, brightness_set, start_captions).
Params: Extracted parameters with null for missing values. Missing params trigger follow-up loop.
Status: complete (fast path), awaiting_param (SLM loop), needs_cloud (cloud path). If user_mode == cloud_always, SLM is bypassed entirely — not loaded into memory. Only activated as offline fallback (cold load ~800ms).
Confidence: 0.0-1.0 score. Below threshold (<0.7) → auto-escalate to cloud even if status is complete.
BG dispatch: bg_dispatch: "start_vision_continuous" triggers background agent without blocking main pipeline.
💡 KV Cache — Why It Matters
KV (Key-Value) cache stores the attention matrices from the SLM’s transformer layers during inference. When the user provides a follow-up answer (e.g. SLM asks “Who?” and user says “Mom”), the KV cache means the SLM doesn’t reprocess the original context — it only processes the new token (“Mom”) against the cached attention state. This cuts follow-up latency from ~800ms to ~400ms.
SLM Follow-up
KV stays warm on-device. ~400ms per follow-up turn. Cleared after pipeline completes.
Cloud Handoff
KV snapshot serialized as context summary → sent to cloud LLM. Cloud avoids re-doing SLM’s work.
Memory Budget
~120MB for KV cache. Evicted by Resource Manager when idle to free GPU memory for vision models.
⚖️ SLM Handles vs Escalates
✅ Handles Locally (~70-80% of requests)
• “Call Mom” → classify + resolve shortcut + execute
• “Set brightness to 70%” → classify + extract param + execute
• “Start live captions” → dispatch BG agent
• “What time is it?” → device clock + TTS
• “Take a photo” → camera shutter + earcon
• “DND on” → system toggle
• Morning brief routine → multi-step deterministic
🌐 Escalates to Cloud (~20-30%)
• “What’s on my schedule and should I leave now?” → multi-tool reasoning
• “Describe what I’m looking at” → vision + scene reasoning
• “Draft a message to my team about standup” → creative generation
• “Compare these two restaurants” → search + analysis
• “Help me plan my commute” → multi-step + external APIs
• Ambiguous: low confidence (<0.7) on any classification
• Unknown intent: not in the ~100 trained classes
StateGraph — Orchestration Engine
⚡ DETERMINISTIC ROUTING + TOOL EXECUTION — LangGraph-compatible, pure Dart
💡 Why
StateGraph is the orchestration engine that manages the pipeline: Stage 3 (Think), Stage 4 (Do), Stage 5 (Deliver), Stage 6 (Maintain). It receives unified input context (text + files + media + links + memory) after Stage 1-2, routes through SLM or cloud for intelligence (Stage 3), manages tool execution (Stage 4), coordinates output + persistence (Stage 5), and oversees background maintenance subsystems (Stage 6). Loop-backs between stages, interrupt handling, and BG agent promotion all flow through StateGraph edges. Unlike LLM-driven routing (where the model decides what to do next), StateGraph is fully deterministic — same input always produces same routing decision. This makes the system predictable, testable, and debuggable.
⏱️ When
Receives Input (first)
StateGraph receives unified context (text + file/media/link refs + memory tiers + device state) after Stage 1-2. Also receives loop-backs from Stage 4/5 and BG agent results. First node checks user_mode: normal → SLM, cloud_always → LLM Gateway.
Tool Execution
After routing, StateGraph also handles tool execution. It maps the intent to a tool from the 121-tool MCP registry, acquires resources via Resource Manager, executes, and collects results.
Cloud Tool Calls
During cloud agentic loop, when cloud LLM requests a tool_call, StateGraph executes it locally and returns the result to cloud. StateGraph is the local executor for both SLM and cloud paths.
BG Agent Dispatch
Two sources: (a) SLM/cloud includes bg_dispatch → Stage 4 starts new finite BG agent. (b) User interrupts current run and backgrounds it → promoted to BG agent with serialized state. Both managed by Agent Manager.
⚙️ How It Works
🛠️ Architecture
Pattern: Dart port of LangGraph. Directed graph with typed state, conditional edges, and node functions. Pure Dart — no Python, no FFI, no network dependency.
Reference: github.com/langchain-ai/langgraph — core concepts ported: StateGraph class, conditional edges, typed state channels, node functions, graph compilation. Adapted for mobile: synchronous execution (no async graph traversal overhead), Dart generics for type safety, Resource Manager integration at execution nodes.
Nodes: Processing steps organized by stage. Stage 3: check_mode, classify/SLM, cloud_gateway, extract_params, route_decision. Stage 4: resource_mgr, execute_tool, bg_agent_startup. Stage 5: tts_output, content_delivery, memory_write, journal_log, loop_check. Each node transforms state and returns updated state.
Edges: Conditional transitions based on state values. if (state.mode == "cloud_always") goto cloud_gateway; else if (state.status == "complete") goto execute; else if (state.status == "awaiting_param") goto ask_user;
State: Typed Dart object carrying the full pipeline context: input text, SLM output, memory context, tool results, conversation history, resource state.
Latency: Graph traversal itself is ~0.1ms (pure in-memory logic). Total pipeline time is dominated by SLM inference and tool execution, not routing.
🔧 Tool Execution Runtime
Registry: 121 MCP-described tools × 7 domains. Each tool has MCP schema (input params, output format, required resources, priority tier).
Resolution: Intent call_make (from SLM or cloud LLM) → registry lookup → TelephonyTool.makeCall(). Direct mapping, no ambiguity.
Resources: Resource Manager runs on every execution — even when no HW is needed, it provides device context (battery level, signal strength, thermal state) that may be useful for further processing. When tools require glasses/phone system resources, it acquires, shares buffers, or queues per resource policy.
Error handling: Tool failures don’t crash the pipeline. Failed tools return error state → StateGraph routes to error recovery (retry, fallback tool, or TTS error message to user).
Parallel: Independent tools can execute in parallel (e.g. fetch calendar + fetch weather). StateGraph collects results via Dart Futures.

LLM Gateway — Cloud Intelligence Router
🔀 LITELLM-PATTERN CLONE — unified multi-provider API, pure Dart
💡 Why
Cloud LLM providers go down, change pricing, throttle requests, and deprecate models. The LLM Gateway abstracts all of this behind a single unified API so the rest of the pipeline never knows or cares which provider is actually responding. If Anthropic is slow, it falls back to OpenAI. If OpenAI is down, it tries Gemini. This gives the system resilience (no single point of failure), cost control (route cheaper for simple tasks, premium for complex), and vendor flexibility (swap providers without touching pipeline code).
⏱️ When
SLM Escalates
When SLM outputs needs_cloud, StateGraph routes through the gateway. SLM’s KV snapshot + classification passed as context.
Cloud Loop Iterations
Each cloud agentic loop iteration (tool_call result, follow_up answer) goes back through the gateway. Same provider kept for session coherence when possible.
BG Cloud Tasks
Background agents needing cloud AI (vision processing, long transcription, document analysis) also route through the gateway for consistent provider management.
Never Used
Fast path and SLM loop (when SLM handles everything) never touch the gateway. ~70-80% of requests avoid cloud entirely.
⚙️ How
🛠️ Architecture
Pattern: Dart port of LiteLLM (Python open-source unified LLM proxy). Same concept — one API call, multiple backends — but implemented natively in Dart for Flutter. No Python dependency, no proxy server.
Reference: github.com/BerriAI/litellm — core logic ported: provider abstraction, fallback chains, retry with backoff, cost tracking, model aliasing. Adapted for mobile constraints (battery, connectivity).
Unified API: All providers expose the same Dart interface: gateway.complete(messages, tools, model_preference). Pipeline code never imports provider-specific SDKs.
Latency: Gateway routing decision ~1ms. Total latency dominated by network + provider inference time (1-10s depending on model/task).
🔁 Provider Chain + Policies
Fallback chain: Local LLM → Anthropic Claude → OpenAI → Google Gemini. Each attempt has configurable timeout + retry count.
Cost budgets: Per-user daily/monthly token caps. Route to cheaper models for simple tasks, premium for complex reasoning. Hive caches routing scores for provider speed/cost.
Model selection: Task-aware routing. Vision tasks → Gemini Flash. Complex reasoning → Claude Sonnet. Quick summaries → cheapest available. Configurable via policy rules.
Offline fallback: Local LLM (Qwen3-0.6B itself in extended mode or future larger local model) attempted first when network is unavailable. Degrades gracefully.
Streaming: Supports server-sent events (SSE) streaming for long responses. First token latency prioritized — TTS can start speaking while cloud is still generating.
🔄 Gateway Flow
1. StateGraph routes needs_cloud → gateway receives: messages + tools + SLM context
2. Gateway checks Hive cache → if exact match with valid TTL, return cached response (skip cloud)
3. Select provider: check policy rules → check provider health → check cost budget → pick best available
4. Call provider API (streaming if supported) → if timeout/error, try next in fallback chain
5. Parse response → normalize to unified format → cache in Hive → return to StateGraph
6. Log: provider used, latency, tokens consumed, cost → journal + Sentry metrics
Resource Manager — Hardware Arbitration
📦 RUNS EVERY PIPELINE EXECUTION — context provider + hardware arbiter
💡 Why
The glasses and phone have shared resources — camera, mic, speaker, BLE bandwidth, GPU memory, display overlay. Multiple consumers compete: the main pipeline, background agents, and system processes. Without arbitration, two agents grabbing the mic simultaneously would produce garbage. The Resource Manager always runs on every pipeline execution (Stage 4). Even when no HW is needed, it provides device context (battery level, thermal state, signal strength) that enriches the SLM/LLM’s decision-making. When tools do need hardware, it acquires, shares, queues, or denies based on priority rules.
⏱️ When
Every Execution (context)
Provides battery %, signal strength, thermal state, active BG agents, available GPU memory. This context is injected so SLM/LLM can make informed decisions (e.g. skip heavy vision if battery <10%).
Tool HW Requests (acquire)
When a tool needs glasses/phone system resources (camera for photo, mic for recording, speaker for TTS, BLE for file transfer), Resource Manager acquires, shares from buffer, or queues.
Contention Resolution
BG Vision using camera + user says “take a photo” → share frame buffer (no conflict). But user says “record video” → exclusive lock needed → BG Vision paused.
GPU Memory Management
SLM KV (~120MB) + Whisper (~75MB model, ~200MB loaded) consume GPU. When vision model needed, evict idle KV cache. Resource Manager tracks and arbitrates GPU budget.
⚙️ How — 6 Managed Resources
📷 Camera
buffered · shareable
BG vision → frame buffer at 2fps. New photo/vision requests read from buffer. Exclusive only during video recording.
🎙️ Microphone
priority-based
Pipeline STT gets priority over BG captions. Time-slice when both need it. Wake word detection always active.
🔊 Speaker
queued
TTS output queued. Priority interrupts (incoming calls, SOS) preempt current audio. BG notifications lower priority.
📡 BLE Bandwidth
allocated
Audio stream gets guaranteed allocation. Sensor data + file transfers share remaining. Throttle non-critical in low-signal.
🧠 GPU/NPU
managed
SLM KV ~120MB + Whisper ~200MB loaded. Vision models loaded on-demand. Evict idle KV cache to free memory.
💻 Display
layered
Overlay system with layers: captions + notifications + navigation. Multiple BG agents write to different layers simultaneously.
Toolkits — 121 Capabilities × 7 Domains
🔧 WHAT THE ASSISTANT CAN ACTUALLY DO — MCP-DESCRIBED TOOL REGISTRY (121 TOOLS)
💡 Why
The SLM classifies intent, but something needs to actually do the thing. The toolkit is a registry of 121 MCP-described tools across 7 domains (118 core + 3 subsystem: lora_train, memory_consolidate, cloud_backup_now). Tools use the Model Context Protocol (MCP, Anthropic open standard) for schema description — any MCP-compatible LLM understands natively, zero adapter code when switching cloud providers. The SLM outputs an intent, StateGraph maps it to the right tool, Resource Manager checks HW availability, and the tool executes via native platform APIs. Adding new capabilities is just adding a new MCP tool class — no pipeline changes needed.
⚙️ How — Tool Structure
🔬 Typed Schema (per tool)
Each tool is a Dart class with MCP-compatible schema: name (MCP tool identifier), description (natural language for LLM understanding), input_schema (JSON Schema typed params), output_schema (structured result), required resources (which HW from Resource Manager), priority tier (P0 emergency → P3 low). MCP format means any compatible LLM can discover and invoke tools without custom adapters. Reference: Anthropic MCP standard.

P0 (14): Emergency, SOS, call answer, power off — never fail, never wait.
P1 (38): Core daily — calls, messages, brightness, volume, timer, photo.
P2 (35): Enhanced — video calls, smart home, calendar, navigation.

Agent Manager — Lifecycle + Resource Control
🔔 UNIFIED AGENT LIFECYCLE — standalone component
💡 Why
Agents consume real device resources — RAM, CPU cycles, battery, network bandwidth. On a phone paired with glasses, resources are already constrained. Without centralized management, agents would compete uncontrolled, drain battery, overheat the device, or exceed platform background execution limits (iOS is particularly strict). The Agent Manager provides a single point of control for all agent lifecycles — main pipeline runs (interrupt handling, pause/resume, state serialization), dispatched background tasks, and promoted interrupted runs, local and cloud alike.
⚙️ How It Works
📱 Local Agent Management
Default max: 4 concurrent (dispatched + promoted combined). Adjustable at runtime based on device capability assessment (RAM available, CPU cores, thermal headroom).
Platform compliance: iOS BGTaskScheduler limits, Android WorkManager constraints, battery optimization whitelist status. Manager adapts to platform-specific restrictions automatically.
Full control: pause (serialize state, free resources), throttle (reduce execution frequency), terminate (cleanup + notify user via TTS). All actions logged to journal.
Timeout: Each local agent has a configurable timeout. Default varies by task type. On expiry: terminate + TTS notify + journal log.
☁️ Cloud Agent Management
Tier-gated max: Free: 0 · Standard: 2 · Plus: 4 · Ultimate: ∞ (fair-use soft limit + throttle). Depends on available API keys and LLM provider rate limits/capabilities.
Long-running: Hours or days allowed. Flight monitoring, package tracking, extended research tasks. No forced timeout — optional user-set limit only.
Local channel control: Manager can pause/throttle the local connection (WebSocket polling frequency, push listener) to save device battery. The cloud task itself is never terminated for device resource pressure.
Termination: Only by user request, task completion, or user-set timeout. Manager sends cancel signal to cloud endpoint.
🔁 Context Rollover — Agent Session Continuity
When an agent’s context window nears capacity (~80%), Agent Manager triggers a rollover to prevent quality degradation (context rot). The agent compacts its own context, a new session spawns with the compact summary, and the task continues transparently. This is the industry-standard pattern used by Anthropic, OpenAI, Google ADK, Manus, and Forge Code.
Rollover Lifecycle
1. Monitor: Agent Manager tracks token count for all active agents (main pipeline + BG). Threshold: ~80% of model context window.
2. Signal: Agent Manager sends rollover instruction to the agent. If mid-tool-execution → complete current tool call first.
3. Compact: Agent consolidates its own context using the same model currently running it. SLM agent → structured template-based compaction (factual, no hallucination). Cloud agent → freeform structured summarization (sections: decisions, state, pending tasks, files touched).
4. Spawn: Agent Manager creates new session with compacted context as initial state. Old session archived. Agent registry updated with new session_id.
5. Continue: Agent resumes task seamlessly from compacted context. Transparent to user.
Compaction Strategy
SLM agents: Structured template-based extraction (same templates as storage consolidation). Extractive approach inspired by LLMLingua-2 and Factory.ai — token classification, not freeform generation. Zero hallucination risk.
Cloud agents: Freeform structured summarization. Follows Anthropic/Manus schema: decisions made, current state, pending tasks, files touched. Higher quality since cloud models have capacity for nuanced summary.
Edge cases: Cloud agent + cloud-less toggle ON → SLM handles rollover (may lose nuance). Privacy mode → rollover works (RAM-only context, compact also RAM-only). Agent fails to respond within 5s → Manager force-terminates, spawns new session with last-known state.
Industry pattern sources: Anthropic SDK (compaction_control), Google ADK (SlidingWindowCompactor), Manus (3-tier compaction → summarization → isolation), Factory.ai (anchored structured preservation), Forge Code / OpenCode (configurable threshold compaction).
📋 Agent Registry
Tracked per agent: run_id, parent_run_id (if promoted), type (local/cloud), source (dispatched/promoted), status (running/paused/throttled), start_time, resource consumption (RAM, tokens), task description.
Dependency graph: Parent→child relationships tracked to prevent circular dependencies. If agent A was promoted from a run that was itself triggered by agent B’s result, and agent B depends on agent A, the cycle is detected and blocked.
User visibility: Agent list available via voice command (“What’s running in the background?”) or companion app dashboard. User can pause, resume, or terminate any agent.
Pipeline Interrupt — Run Lifecycle Management
⚡ INTERRUPT HANDLING + STATE SERIALIZATION + RUN PROMOTION
💡 Why
AI glasses are real-time, always-carried devices. Unlike desktop apps, the user can’t just switch tabs — they’re wearing the device while living their life. An incoming call, a new file share, or a sudden “Hey Humbl” can happen at any time, including mid-pipeline. Without proper interrupt handling, the system either ignores new input (frustrating) or crashes the current task (data loss). The interrupt system lets the pipeline gracefully handle competing demands — pause, resume, background, or terminate runs while preserving state.
⚙️ How It Works
🚨 Interrupt Detection
Listener: A lightweight interrupt listener runs in parallel with every active pipeline. Monitors: OS callbacks (incoming call, memory warning), hardware triggers (AI button, wake word VAD), and app events (share sheet, push notification action).
Priority: System interrupts (OS-level) bypass user choice and immediately pause/terminate. User/event interrupts pause the current run and trigger a mini decision agent.
Latency: Interrupt detection <50ms. State serialization ~100-300ms depending on KV cache size.
💾 State Serialization
What’s saved: Full ConversationState object — input context, SLM KV cache snapshot, cloud conversation history, partial tool results, memory refs, current stage + node position, run_id + parent chain.
Where: Hive box (fast, structured) for active pause state. SQLite for longer-term promoted BG agents. Both keyed by run_id.
Resume: Deserialize state → restore to exact node position in StateGraph → continue execution. KV cache reloaded (warm ~400ms if SLM, cloud context re-sent if cloud).
🔔 Run Promotion (to BG Agent)
When: User chooses “Background” during active execution interrupt. Only available when run is in Stage 3 (reasoning) or Stage 4 (executing) — not during mid-follow-up (needs user input).
Process: Serialize full state → register as BG agent with unique run_id + parent_run_id → Agent Manager takes ownership → counts toward concurrency limits (local or cloud based on execution type) → continues execution independently.
Result: When promoted BG agent completes, result enters Stage 3 input queue. Stage 5 notifies user via TTS. User can inspect result or continue from where it left off.
Anti-patterns blocked: Circular deps (run_id parent chain checked), nesting overflow (concurrency limit enforced), stale runs (timeout applies).

Memory System — 4-Tier Persistent Intelligence
🧠 MAKES THE ASSISTANT PERSONAL — remembers, learns, adapts · IMPORTANCE-SCORED EVICTION · NO TTL WINDOWS
🎯 Architecture principle: Hive Cache (performance layer, sits IN FRONT of pipeline, caches computed results) is NOT part of the memory system. Memory tiers T1–T4 store persistent knowledge in SQLite. Hive caches LLM responses, RAG search results, vision outputs, and TTS audio for sub-millisecond repeat access. See Hive Cache deep dive for full details. Memory eviction is importance-scored (not time-windowed) — high-importance memories survive indefinitely regardless of age.
💡 Why
Without memory, every interaction starts from zero — the user must repeat preferences, explain shortcuts, re-establish context. The 4-tier memory system gives the assistant personal context (who “Mom” is, what “my usual brightness” means), conversational continuity (remembering “that restaurant from last week”), and adaptive behavior (auto-learning that the user always chooses Slack for team messages). All stored locally on-device, privacy-first. Industry-standard importance scoring (Mem0, Zep, Letta patterns) ensures critical facts (“allergic to penicillin”) never evict while trivial queries reclaim storage under pressure.
⏱️ When — Memory Lifecycle
📖 READ
When: Context assembly (Stage 2), before SLM processes every request. Hive checked first — if full response cache hit, skip pipeline. For memory reads: T2 Knowledge Store queried for critical+high-importance entries, T3 semantic search (MiniLM embed → sqlite-vec) for contextual recall. All injected into SLM prompt via priority-ordered context assembly. Also forwarded to cloud LLM if escalated.
🔎 LOOKUP
When: During processing, resolving user references. “my usual” → T2 preference. “that restaurant” → T3 vector search. “Mom” → T2 entity index. Latency: T1: 0.1ms (RAM), T2: 1-5ms (SQLite), T3: 10-30ms (vector search).
✎️ WRITE
When: Post-execution, every pipeline run. T1: update session context. T2: increment pattern counters (3+ → auto-graduate to preference), assign importance scores. T3: embed interaction summary via all-MiniLM-L6-v2 (~5-10ms, async), link entity refs. T4: append raw log entry. Hive: cache computed results (LLM/RAG/TTS). Write-through: every T2/T3 write invalidates matching Hive keys.
💡 LEARN
When: Pattern Learner BG agent (async, periodic). Mines T4 logs for repeated behaviors. User says “glasses” for camera 3x → auto-store as default (importance 0.7). DND at 11pm 5 nights → suggest routine. Contact called 3x → offer shortcut. Writes to T2 (prefs + entity index) and T3 (embeddings).
🔁 HANDOFF
When: SLM escalates to cloud. KV snapshot + relevant memory context serialized → sent to cloud LLM. Cloud sees: user prefs (high-importance T2 entries), entity index, recent history, SLM’s partial classification. Cloud can also trigger memory writes (extracted facts → T2/T3 with importance scores).
⚙️ How — 4 Storage Tiers
Tier 1 — Session Context
⚡ ~0.1ms · RAM-only · destroyed on app close
Stores: Current conversation turns, device state snapshot, active intent chain, pending tool results, BG agent references.
Lifespan: One session only. Not persisted to disk. Not synced. Purely ephemeral working data for current interaction.
Fed into: KV cache during context assembly (Stage 2). Recent turns injected into SLM prompt.
Tier 2 — Knowledge Store
~1-5ms · SQLite (sqflite) · persistent · importance-scored
Stores: User preferences (“prefers dark mode”, “allergic to penicillin”), learned patterns (“sets alarm 7am weekdays” after 3+ repeats), contact shortcuts (“Mom” → Sarah Johnson), device defaults.
Entity index: Named entities (person, place, thing) → FK to T2 row IDs + T3 embedding IDs. Enables entity-based lookup: “what did Sarah recommend?” → find all memories referencing “Sarah”.
Each entry: importance_score (0.0-1.0), access_count, last_accessed, source (user_stated | inferred | corrected).
Eviction: Importance-scored. Lowest-importance entries evicted first under storage pressure. No TTL windows.
Tier 3 — Semantic Memory
~10-30ms · sqlite-vec + all-MiniLM-L6-v2 · persistent · importance-scored
Stores: Interaction summaries as 384-dim embeddings. Semantic search for contextual recall. Each entry: importance_score, embedding, summary_text, entity_refs (FK to T2 entity index), created_at, last_accessed.
Search: Hybrid — BM25 keyword (30% weight via FTS5) + vector cosine (70% weight via sqlite-vec). Union merge.
Embeddings: all-MiniLM-L6-v2 (~23MB ONNX int8, FFI, ~5-10ms per embed, runs locally on-device).
Consolidation: Cluster similar (≥0.85 cosine) → merge → reembed. Merged entry inherits MAX importance from sources.
Tier 4 — Audit Log
Background · append-only · SQLite t4_log table · LoRA training source
Stores: Every command, timestamp, latency, result, provider used, tokens consumed. Raw source-of-truth.
Updated: Every pipeline run (always appended). Never modified — immutable audit trail.
Consumers: Pattern Learner BG agent mines this for behavior patterns (graduate into T2). LoRA subsystem reads T4 for fine-tuning training data extraction (post-pipeline, zero coupling).
Size-managed: Old logs → compressed archive. Never evicted, only archived.
⚖️ Importance Scoring — Eviction by Value, Not Time
Every T2 and T3 entry receives an importance score (0.0–1.0) at write time. Eviction targets lowest-importance entries first under storage pressure. High-importance memories survive indefinitely regardless of age. Scores decay slowly if not accessed — a score-0.7 memory not accessed in 30 days might decay to 0.5. Access resets decay. This prevents “importance inflation” while keeping actively-used knowledge fresh.
0.9–1.0 🔴 Critical — Never evicted
User said “remember this”. Medical/safety (allergies, medications, emergency contacts). Identity facts (name, birthday, address). User corrections of previous inferences.
0.7–0.8 🟠 High — Protected pool
Learned preferences confirmed 3+ times. Named entity relationships (“Sarah is my wife”). Recurring patterns (“gym every Tuesday”). Tier-dependent protected pool size.
0.4–0.6 🟡 Medium — Normal eviction eligible
Single-mention preferences. Contextual facts. Inferred patterns (2 occurrences). Recent interactions not yet confirmed as patterns.
0.1–0.3 🔵 Low — Evicted first
One-time queries (“what’s the weather”). Transient context (“I’m at the coffee shop”). Outdated info (superseded by correction). Evicted first under storage pressure.
📈 Tier Differentiation (protected pool size)
Free: Hive 5–10MB, protected pool top 50 (local only), full hybrid consolidation, cloud via BYOK only
Standard: Hive 15–30MB, protected pool top 200, full hybrid SLM consolidation
Plus: Hive 30–60MB, protected pool top 500, full hybrid consolidation
Ultimate: Hive 60–100MB, protected pool top 2000, full hybrid consolidation
🔄 Score Decay Formula
Importance scores decay slowly if not accessed. effective_score = base_score * decay_factor(days_since_access). Access (read or write) resets decay timer. Critical scores (≥0.9) never decay. Prevents stale medium-importance entries from blocking new high-value memories.
🔒 Memory Consistency Model
Within Session
SQLite WAL mode (readers never block writers). Hive box-level locks (Dart isolate-safe). Single Memory Manager singleton serializes importance score updates.
Cache ↔ Storage
Write-through invalidation. Every T2/T3 write → invalidate matching Hive keys (rag, session boxes). Next read → Hive miss → fresh SQLite data.
Cross-Device
Eventual consistency via cloud sync. Conflict resolution: version counter + device_id + timestamp. Higher version wins; same version = latest timestamp wins. T4: no conflicts (append-only, merge by timestamp).
Eviction ↔ Consolidation
Never run concurrently (Memory Manager lock). Consolidation recalculates importance scores atomically per batch. Eviction reads scores but never writes them.
🔁 KV Cache ↔ Memory Integration
KV cache (RAM-only, ~120MB for Qwen3-0.6B) holds the SLM’s active attention state for the current conversation. It is not memory — it is destroyed on app close or context rollover. The critical intersection is Stage 2 context assembly: memory feeds INTO KV cache, and during rollover, important KV discoveries write BACK to memory.
👉 Memory → KV (Context Assembly, Stage 2)
Priority injection order:
1. System prompt + tool definitions (~2K tokens, fixed)
2. T2 critical facts (importance ≥0.9, ~500 tokens)
3. T1 session context (recent turns, variable)
4. T2 high-importance prefs (≥0.7, ~500 tokens)
5. T3 semantic search top-5 (~1K tokens)
6. T2 medium-importance (if space allows)
If projected total >70% context window: skip step 6, reduce T3 to top-3.
👉 KV → Memory (Rollover Writeback)
During context rollover (at ~85% KV usage):
• SLM compacts conversation into summary
• Important discoveries → write to T2/T3 with importance scores
• New session spawns (fresh memory injection + compact summary)
• KV cache rebuilt from scratch
Circular dependency: memory makes KV useful, rollover keeps memory fresh.
⚡ 2-Stage KV Cache Management
Stage A: KV Eviction (~70% usage)
Analyze attention scores across KV pairs. Mark memory-injected tokens (assembly steps 1-5) as PROTECTED. Evict lowest-attention conversation tokens only (old turns). Free ~15-20% context window. Session continues seamlessly — user unaware.
Stage B: Full Rollover (~85% usage)
KV eviction wasn’t enough. SLM compacts conversation into summary. Important discoveries written to T2/T3 with importance scores. New session spawns with fresh memory injection + compact summary. KV cache rebuilt from scratch.
Memory Consolidation & Context Rollover — Long-Term Intelligence
🧠 TWO CONSOLIDATION MODES — STORAGE (IMPORTANCE-SCORED EVICTION) + CONTEXT (AGENT SESSION CONTINUITY)
💡 Why
Two problems grow with usage: (1) Storage pressure — T2/T3 memory fills as the user accumulates interactions. Without consolidation, storage bloats unbounded. The consolidation engine clusters related memories, recalculates importance scores, and compacts into fewer entries while preserving knowledge. Eviction targets lowest-importance entries first — not by age (no TTL windows). (2) Context rot — as an agent’s conversation grows, LLM performance degrades even within token limits (Factory.ai, Manus, and academic research confirm this). Context rollover compacts the agent’s working history before quality drops, enabling seamless long-running tasks.
📦 Mode A — Storage Consolidation (Importance-Scored)
5-stage hybrid pipeline triggered when T2 or T3 storage exceeds >80% of tier’s size limit (NOT calendar-based, NOT TTL-based). Clusters related memories, recalculates importance scores, merges duplicates, and evicts lowest-importance entries. Runs as a low-priority BG agent. All tiers run the same hybrid pipeline. No tier differentiation for consolidation — every user gets full importance scoring + SLM classification + template compaction.
Stage 1 — Embedding Clustering
sqlite-vec cosine similarity + all-MiniLM-L6-v2
What: Groups related memories by semantic similarity. Threshold ~0.85 cosine distance. Detects near-duplicates (e.g., 15 separate grocery trips that all mention the same store).
Output: Clusters of related memory entries + duplicate pairs flagged for merge.
Cost: 0 additional RAM — reuses existing sqlite-vec index and embedding model already loaded for T3 semantic search.
Stage 2 — Importance Scoring
Qwen3-0.6B (SLM) · structured JSON classification
What: SLM classifies each memory (or cluster) as high, medium, or low importance. Structured JSON output ensures parsing reliability.
Factors: Access frequency (how often recalled), recency (last access time), user-explicit saves (starred/pinned items), entity references (contacts, locations, events), emotional valence.
Why SLM works: Classification is a task 0.6B models handle reliably (structured output, bounded categories). No freeform generation = no hallucination risk.
Stage 3 — Template Compaction
Dart template engine + SLM entity extraction
What: Fills structured merge templates from clustered memories. No freeform generation — SLM extracts entities (who, what, where, when, frequency) and Dart code fills pre-defined templates.
Example: 15 grocery trips → {user} shops at {store} {frequency}, prefers {categories}
Why templates: 0.6B models drift and hallucinate in freeform summarization. Templates guarantee factual accuracy. Industry-validated (SimpleMem, Mem0 use similar structured extraction).
Stage 4 — Archive / Prune
SQLite + sqlite-vec · tier-gated destination
High importance (≥0.7): Protected pool. Never evicted unless user explicitly deletes. Cloud backup if tier allows.
Medium importance (0.4–0.6): Normal eviction eligible. Kept while storage permits. Evicted under pressure after low-importance exhausted.
Low importance (0.1–0.3): Evicted first under storage pressure. Original T4 log entry preserved as audit trail. Reclaims storage.
Re-embed: Compacted entries get new 384-dim embeddings via all-MiniLM-L6-v2, replacing the embeddings of the individual source memories in sqlite-vec.
⚙️ Operational Constraints
Trigger: Storage pressure only (not scheduled, not time-based). T2 or T3 usage exceeds >80% of tier size limit → pipeline starts.
Execution: Runs as registered BG agent via Agent Manager. Low priority — yields to foreground, throttles under thermal pressure.
Batch size: 50–100 entries per pass. SLM loaded on-demand if not in memory (cold start ~800ms acceptable for BG).
Additional RAM: 0 — reuses existing SLM, embedding model, and sqlite-vec. No new models or libraries required.
Free tier: Same full hybrid pipeline as all tiers. Protected pool: top 50 entries. Only difference from Standard: no cloud backup.
Privacy mode: No consolidation possible (nothing persisted to disk). RAM-only T1 context cleared on app close.

LoRA Personalization — On-Device Fine-Tuning
🧠 INDEPENDENT SUBSYSTEM · PIPELINE-INVOCABLE VIA TOOL · ZERO INFERENCE OVERHEAD · ALL TIERS · DEVICE IDLE TRAINING
💡 Why
Every user speaks differently — abbreviations, domain vocabulary, command patterns. LoRA (Low-Rank Adaptation) fine-tunes 0.1–1% additional parameters on the frozen base SLM, producing a small adapter (~5–20MB) that personalizes intent recognition, parameter extraction, and domain vocabulary. Training data never leaves the device. The adapter is weight deltas only — meaningless without the matching base model.
🎯 Architecture principle: LoRA is an independent subsystem that runs autonomously (device idle training, automatic data collection from T4). The pipeline can invoke LoRA operations as a tool via StateGraph (e.g., “retrain my model”, “check adapter status”) — but LoRA never blocks or modifies the inference path. Adapters merge at SLM cold start only. No per-inference overhead after weight merge.
⚙️ Subsystem Components
1. Data Collection
Post-pipeline BG micro-task · async
Trigger: T4 log entry written (filesystem/DB notification). Pipeline already finished, response delivered.
Action: Read latest T4 entry → extract {intent, params, correction, outcome} → write to LoRA staging buffer (separate SQLite table).
Cost to user: 0ms (pipeline already done).
2. Training Engine
Device idle window · QLoRA · 10-30 min
BG trigger: Device idle (all conditions met) + staging buffer ≥500 pairs.
FG trigger: User-initiated (“Retrain” in settings).
Process: QLoRA fine-tune on base SLM → adapter .bin (~5–20MB).
Cost to user: 0ms pipeline (separate process, invisible).
3. Validation
Automated quality gate · rollback capable
Test: Run adapter against 20 cached interactions. Compare confidence scores vs. baseline (no-adapter) model.
Pass: Save adapter + flag for loading on next cold start.
Fail: Discard adapter, keep current version. Log failure reason to T4.
4. Adapter Store + Loading
Versioned .bin · merge at cold start only
Store: Versioned adapter files (adapter_v1.bin, adapter_v2.bin). Rollback capability.
Loading: Merge adapter weights into base model at SLM cold start. +50–100ms to cold start (~800ms → ~900ms). 0ms per-inference after merge (weights pre-merged in RAM).
Cloud sync: Latest validated adapter synced to cloud backup. Metadata: version, base_model_hash, training_date, device_id, pair_count. Only adapter synced (~5–20MB), never base model (~400MB).
Device handoff: New device downloads base SLM (app bundle) + restores adapter from cloud → personalized model immediately.

all-MiniLM-L6-v2 — Embedding Model
🔢 TEXT → 384-DIM VECTORS — enables semantic search across memory
💡 Why
When the user says "that restaurant from last week", keyword search fails — the word "restaurant" might not appear in the stored interaction that said "directions to Bombay Bistro". The embedding model converts text into 384-dimensional vectors where semantically similar texts are close together in vector space. "that restaurant" and "Bombay Bistro directions" end up near each other, enabling the memory system to recall the right context. Without embeddings, the system loses ~70% of its contextual recall ability and falls back to rigid keyword matching only.
📋 What It Is
🧠 Model Specs
Model: all-MiniLM-L6-v2 — a sentence-transformer distilled from MiniLM. 6 transformer layers, ~22M parameters.
Output: 384-dimensional float vector per input text. Normalized for cosine similarity search.
Format: ONNX int8 (~23MB on disk). Loaded via Dart FFI to ONNX Runtime. No Python, no TensorFlow, no server.
Latency: ~5-10ms per embedding on mobile CPU. Fast enough for real-time use in the pipeline.
Quality: Top-tier for its size class. Outperforms models 5-10x larger on semantic similarity benchmarks. Optimized for short texts (sentences, commands, summaries).
License: Apache 2.0. No usage restrictions. Hosted on HuggingFace: sentence-transformers/all-MiniLM-L6-v2
🔧 Tech Stack
Runtime: ONNX Runtime via Dart FFI. Platform-specific: Core ML delegate (iOS), NNAPI delegate (Android) for hardware acceleration when available. CPU fallback always works.
Tokenizer: WordPiece tokenizer (same as BERT family). Bundled as JSON vocab file (~230KB). Max 256 tokens per input.
Memory: ~23MB model weight (int8 quantized) + ~5MB runtime overhead. Loaded once on app start, stays resident. Shared across all embedding operations.
Vector store: Embeddings stored in sqlite-vec extension (float32 arrays). Cosine similarity search via sqlite-vec's built-in knn() function.
Hybrid search: sqlite-vec (70% vector cosine) + FTS5 (30% BM25 keyword). Results merged by union with weighted scoring.
⏱️ When It Runs
Memory READ (T3 search)
During context assembly, when the query needs semantic matching. User input → embed via MiniLM → cosine search in sqlite-vec → retrieve relevant past interactions. Only triggered when T1/T2 don't have the answer.
Memory WRITE (T3 store)
Post-execution, async. Interaction summary text → embed via MiniLM (~5-10ms) → store vector in sqlite-vec. Runs in background, never blocks pipeline response.
Pattern Learner (T4 → T3)
BG agent mines daily logs (T4), identifies graduated patterns, writes embeddings to T3 for future semantic recall. Runs periodically during idle.
Cloud Sync (T3 → Supabase)
T3 embeddings synced to Supabase PostgreSQL (pgvector). Enables cross-device memory restore. Embeddings batched every 15min.
Hive Cache Miss
When Hive RAG cache doesn't have a recent result for this query, the full embed → search pipeline runs. Result cached in Hive for next similar query (TTL 5-15min).
NOT Used For
T1 hot cache lookups (key-based), T2 KV prefs (SQL queries), T4 log writes (append-only), SLM inference (SLM has its own model). Embeddings are purely for T3 semantic memory.
🔄 End-to-End Embedding Flow
WRITE path (async, post-execution):
  Interaction summary text → WordPiece tokenizer (→ token IDs) → MiniLM ONNX inference (~5-10ms) → 384-dim float32 vector → sqlite-vec INSERT → Hive RAG cache invalidated

READ path (context assembly):
  User query → check Hive RAG cache → if miss: WordPiece tokenize → MiniLM embed (~5-10ms) → sqlite-vec knn(query_vec, k=5) → cosine similarity top-5 results
  Parallel: FTS5 BM25 keyword search on same query text
  Merge: vector results (70% weight) ∪ keyword results (30% weight) → deduplicate → ranked context → inject into SLM prompt
  Cache merged result in Hive (TTL 5-15min) for next similar query
Hive Cache — Local Redis Equivalent
⚡ PURE-DART NoSQL KV — sits in front of pipeline, not inside memory tiers
💡 Why
Cloud LLM calls cost money and time (1-10s). Vector search costs compute (~10-30ms). Vision decoding costs CPU. If any of these produced a result recently, why redo the work? Hive acts as a local Redis — fast key→value lookups with TTL-based expiry, sitting in front of the pipeline. It intercepts requests before they reach SLM or cloud, and caches results from every subsystem. This saves API costs, reduces latency to near-zero for repeat/similar queries, and keeps cached responses available offline. Hive is not part of the 4-tier memory system — it's a performance layer that sits alongside all of them.
🔗 How Hive Ties to Every Subsystem
🧠 SLM + Cloud LLM Responses
What: Full pipeline results — SLM outputs + cloud LLM responses + tool execution results.
Key: Hash of normalized user input + classified intent + params. Similar queries hit same cache entry.
TTL: Varies by type — weather/time-sensitive: 15-30min, factual: 24h, device commands: no cache (always execute fresh).
Flow: Request → Hive lookup → if hit, skip SLM/cloud entirely → TTS cached result → memory write still happens (T4 log + T1 update).
🔍 Memory System (T3 Vector Search)
What: RAG vector search results from sqlite-vec. The embedding + similarity search is the most expensive local operation (~10-30ms).
Key: Query embedding hash. "that restaurant" → same vector → same cached T3 result.
TTL: Short (5-15min). Context changes frequently — new interactions shift what's relevant.
Invalidation: Memory WRITE to T3 (new embedding stored) triggers invalidation of affected Hive RAG entries.
🔀 LLM Gateway (Provider Routing)
What: Provider health + latency + cost scores from recent cloud calls. Which provider was fastest? Cheapest? Had errors?
Key: Provider name + model + region.
TTL: Rolling window (last 10min of scores). Stale data means bad routing decisions.
Used by: LLM Gateway's provider selection logic — pick fastest provider that's under budget and healthy.
👁️ Vision Pipeline (ML Kit)
What: Google ML Kit decode results — QR code content, OCR text, barcode data, object detection labels.
Key: Frame hash (perceptual hash of camera frame region). Same QR code in view → cached result.
TTL: Session-length for static objects (QR codes), very short (seconds) for scene objects.
Why: BG Vision agent processes at 2fps. Without cache, same QR code gets decoded 120 times per minute.
🗣️ TTS Template Cache
What: Pre-synthesized Piper audio clips for common response patterns — "Calling [name]", "Brightness set to [X]%".
Key: Template ID + slot values. "calling_mom" → cached audio bytes.
TTL: Long-lived (days). Only invalidated if Piper model or voice changes.
Why: Piper synthesis takes ~100-200ms. Cached templates respond in ~30-50ms (3-6x faster).
📋 Session State + BG Agent Buffers
What: Active conversation context, BG agent intermediate state, Resource Manager device snapshot, current user session data.
Key: Session ID, agent ID, resource type.
TTL: Session-scoped. Cleared on session end or app background timeout (configurable, default 30min).
Why: Between pipeline runs within a session, state must persist without SQLite writes for every micro-change.
🧠 Memory Fast-Access Layer (T1/T2 hot data)
What: Frequently accessed memory data from T1 (last 5 commands, device state) and T2 (top contact shortcuts, most-used preferences). Hive mirrors the "hottest" rows from SQLite so context assembly doesn't hit disk on every pipeline run.
Key: Memory tier + record ID. mem:t1:last_commands, mem:t2:shortcut:Mom
TTL: Matches the underlying SQLite write frequency. T1 entries expire after each pipeline run (refreshed). T2 entries expire on next T2 write or preference change.
Sync: Hive → SQLite is one-way read cache. SQLite is always source of truth. Memory WRITE updates SQLite first, then refreshes Hive mirror. On cold start, Hive populates from SQLite top-N rows.
Why: SQLite in-memory page cache is ~0.1ms, but Hive HashMap lookup is ~0.01ms. For the hottest data accessed on every single pipeline run (T1 context), that 10x difference adds up across thousands of daily requests.
💾 Where + How Hive Persists
📦 On-Disk Storage
Location: App documents directory — /data/data/com.humbl/hive/ (Android) or app sandbox (iOS).
Format: Hive binary box files. One box per cache domain: llm_cache.hive, rag_cache.hive, vision_cache.hive, tts_cache.hive, routing.hive, session.hive.
Size: ~5–100MB total (tier-dependent: Free ~5–10MB, Standard ~15–30MB, Plus ~30–60MB, Ultimate ~60–100MB). Performance cache for sub-millisecond repeat access. NOT a memory tier — caches computed results only. Bounded by max entry count per box (configurable, default 500–2000 entries per box).
Persistence: Lazy write — data is in-memory first, flushed to disk asynchronously. Survives app restarts. Lost on app uninstall.
♻️ Lifecycle + Eviction
TTL expiry: Each entry has a TTL stamp. Expired entries skipped on read, purged on periodic compaction (every 100 writes or on app cold start).
LRU eviction: When a box exceeds max entries, least-recently-used entries evicted first. Cache-miss is never fatal — pipeline just runs normally.
Manual invalidation: Memory T2/T3 writes trigger targeted cache invalidation. User changes a preference → any cached LLM response that used that preference gets evicted.
Cold start: Hive loads from disk on app launch (~10-50ms). Warm cache available immediately. No network needed.
🛠️ Implementation
Library: Hive v2 (Apache 2.0) — pure Dart, zero native deps, zero platform channels. Works identically on iOS + Android.
Memory model: In-memory HashMap backed by lazy disk flush. Read: O(1) hash lookup (~0.01ms). Write: O(1) insert + async disk write.
Relationship to memory: Hive is NOT a memory tier — it's a fast-access layer that sits alongside them. Memory tiers (T1-T4) store persistent knowledge in SQLite (source of truth). Hive caches computed results (LLM responses, RAG searches) AND mirrors hot memory data (T1 recent commands, T2 top shortcuts) for sub-millisecond reads. Memory WRITE always updates SQLite first, then refreshes Hive mirrors. Hive is "have I answered this before?" + "what's the hottest data I'll need next?"; memory is "what do I know about this user?"
Not synced: Hive is local-only, never synced to Supabase. It's ephemeral acceleration — rebuilt naturally from usage. Memory tiers sync to cloud; cache does not.

Content Lifecycle — Files, Media, Links, Artifacts
📦 INPUT → PROCESSING → STORAGE → MEMORY REF → CROSS-RUN RETRIEVAL → VERSIONING
💡 Why
Users interact with content across multiple sessions — “summarize this PDF” today, “pull the revenue numbers from that budget doc” tomorrow, “add a cover page to it” next week. Without a content lifecycle, each run starts from scratch: re-upload, re-parse, re-analyze. The content lifecycle ensures every file, media item, link, and AI artifact is stored once, referenced in memory, version-tracked, and instantly retrievable across any future agent run.
⚙️ Lifecycle Stages
1. Input
Content enters via event trigger payload (file share, camera capture, link paste, QR scan) or is produced by agent (AI artifact, fetched doc). Raw reference attached at Stage 1-2. Memory check: if previously processed, cached content attached immediately.
2. Processing
Stage 3 decides what tools to call. Stage 4 executes: file_parse (PDF/DOCX/CSV), ocr_scan (ML Kit), link_fetch (metadata + content), image_analyze, etc. Result is structured output (text, tables, objects detected, etc.).
3. Storage
Local file: persists at known path. Hive cache: parsed content for fast re-access (TTL-based). Supabase: optional cloud sync for multi-device access.
4. Memory Reference
T2 KV: quick lookup by name/type (file:budget_q3). T3 vector: semantic search via MiniLM embedding (“that financial doc from last week”). T4 log: audit trail of all operations.
5. Cross-Run Retrieval
Future run references content → memory check at Stage 1-2. T2 KV hit → attach metadata + Hive cached content. T3 vector match → semantic match for vague references. Hive expired → re-read from local file path, re-cache.
6. Versioning
Modifications create new version — original preserved. T2 KV latest pointer updated, version history array maintained. User can request any version: “give me the original” or “undo that change.”
📄 Content Types
📄 Files
PDF, DOCX, CSV, TXT, XLSX. Parsed to text + structure. Versioned on modification.
📷 Media
Images, audio, video. ML Kit analysis cached. Thumbnails stored in Hive.
🔗 Links
URLs with fetched metadata (title, OG tags). Full content optionally cached. Refreshable on demand.
📦 AI Artifacts
Generated summaries, code, structured data, reports. Stored as files. Versioned if iterated.

Backend — Self-Hosted Supabase
🌐 SYNC + BACKUP + AUTH + SESSIONS + TELEMETRY — cloud layer for on-device-first architecture
💡 Why
The AI pipeline runs entirely on-device, but users need accounts (login, device pairing), backup/restore (new phone, factory reset), cross-device sync (glasses paired with different phones), session tracking (usage analytics, billing), cloud memory persistence (preferences survive device loss), and operational observability (crash reports, telemetry). Supabase provides all of this as a self-hosted PostgreSQL stack — no Firebase, no vendor lock-in, full data ownership.
⚙️ Core Infrastructure
🔐 Auth
JWT + OAuth (Google, Apple). User registration, login, device pairing tokens, session refresh. Row-Level Security (RLS) ensures users only access their own data.
🗄️ PostgreSQL + pgvector
Source of truth for synced data. User profiles, device metadata, T2 prefs, T3 embeddings (pgvector), session logs, aggregated analytics. Realtime subscriptions push config changes.
📁 Storage (S3-compatible)

Counts toward tier storage quota (0/1/5/20 GB). Quota enforcement: 80% notify, 95% throttle (skip low-importance T3), 100% stop sync.
⚡ Edge Functions (Deno)

👤 User Management + Sessions
🔑 Account + Device Pairing
Registration: OAuth (Google/Apple) or email+password. Account in Supabase Auth. User ID (UUID) is FK for all tables.

💥 Failure Modes & Graceful Degradation
BLE Disconnect
Queue events on both Agent + Connect. Exponential backoff reconnect (1s → 2s → 4s → 8s). Auto-resume on reconnect. If >30s: TTS “glasses disconnected” + companion app notification. If >5 min: stop audio routing, switch to phone-only mode.
Phone Offline (no network)
Local-only AI: SLM handles intent + local tools. Cloud tools unavailable — TTS “I’m offline, I can do [local capabilities]”. Queue logs + memory writes for sync when network returns. Cloud BG agents: pause polling, resume on reconnect.
Cloud LLM Outage
LLM Gateway: auto-failover across 3+ providers (latency-ordered). All providers down: fall back to SLM on-device for basic tasks. Cached cloud responses served from T1 Hive. TTS “Cloud AI is unavailable, using local mode”.
PostgreSQL Down
Read-only mode via streaming replicas. New writes queued locally in SQLite + Hive. Edge Functions return 503 for writes, 200 for reads. Auto-failover to replica as primary: <5 min. Full restore from WAL: <60 min.
SLM Model Corruption
Checksum validation on load. Corrupt: re-download from Supabase Storage (~400MB, 1-5 min on Wi-Fi). Meanwhile: cloud always mode handles all AI. TTS “Updating local AI model”.
Memory Pressure (OS kill risk)
Resource Manager cascade: evict T1 cache → evict KV cache → unload SLM → throttle BG agents → terminate BG agents. Each step frees 5-520MB. State serialized before eviction. Recovery: reload from disk on next request.

📖 Terminology: Data Saving = persist to local storage (SQLite T2–T4). Survives app restart. Data Backup = sync local storage to cloud (Supabase). Full device mirror for migration/handoff. Cross-device + disaster recovery. Hive Cache = performance cache layer (NOT memory). Sits in front of pipeline. Caches LLM responses, RAG results, TTS audio. Tier-dependent size: 5–100MB. Importance Scoring = every T2/T3 memory entry scored 0.0–1.0. Eviction targets lowest-importance first under storage pressure. No time-based TTL windows. Protected Pool = tier-dependent count of high-importance entries (≥0.7) exempt from eviction. Free: 50 (local only), Standard: 200, Plus: 500, Ultimate: 2000. Protected pool entries are included in cloud backup on Standard+ tiers. Memory Consolidation = hybrid pipeline (embedding clustering + SLM classification + template compaction) triggered at >80% T2/T3 storage. Deduplicates, merges, recalculates importance. Same pipeline for all tiers. Context Rollover = automatic agent session continuation when context window nears capacity. Agent Manager monitors, agent compacts its own context, new session spawns with compact summary. Saving is prerequisite for backup — can save without backing up, cannot back up without saving.

🧠 Memory Consolidation Engine (All Tiers)
4-stage hybrid pipeline for long-term memory compaction. Runs as low-priority BG agent when storage pressure detected. Same pipeline across all tiers — no tier differentiation.
// storage consolidation pipeline
1. Embedding Clustering — sqlite-vec cosine sim + all-MiniLM-L6-v2 (threshold ~0.85) → group related memories
2. Importance Scoring — Qwen3-0.6B classifies high/medium/low (structured JSON). Factors: access frequency, recency, user-explicit saves, entity refs
3. Template Compaction — Dart template engine + SLM entity extraction. No freeform generation = zero hallucination
4. Archive/Prune — High → keep fast. Medium → local archive. Low → prune. Re-embed compacted entries.
Additional RAM: 0 — reuses existing SLM, embedding model, sqlite-vec. Batch: 50–100 entries/pass. SLM loaded on-demand (cold start ~800ms acceptable for BG). Yields to foreground, throttles under thermal pressure. Industry-validated pattern (SimpleMem, Mem0, LangMem use similar hybrid approaches).

🔄 Data Sync Topology — Instance Coordination
Shared Data Store
PostgreSQL (primary): Single source of truth for all user data (T2/T3/T4 cloud mirrors, auth tokens, billing). All instances connect via Supavisor pooler (not direct PG connections). WAL-based streaming replication to read replicas.
Object Storage (R2/S3): Shared blob store for media, LoRA adapters, content cache. All instances read/write same bucket via presigned URLs.
No shared Redis: On-device Hive cache eliminates server-side caching. Redis used only for rate-limiting + ephemeral session tokens at 25K+ DAU.
Cross-Instance Sync
Conflict resolution: version_counter + device_id + timestamp (last-write-wins). Each device owns its data partition — no cross-device write conflicts at sync layer.
Consistency model: PG WAL mode for write-through. Supabase Realtime (WebSocket) for push notifications across instances. Eventual consistency between primary ↔ replicas (~10-100ms lag).
Instance affinity: Users pinned to closest PG replica via Supavisor geo-routing. Writes always route to primary. Read-after-write consistency via connection stickiness.
[Device A] → delta sync → [Supavisor Pooler] → [PG Primary] ← WAL replication → [PG Replica 1 (Mumbai)] + [PG Replica 2 (Singapore)]
[Device B] → sync read → [Supavisor] → geo-route → [nearest replica]   |   [Device B] → sync write → [Supavisor] → [PG Primary]
⚖️ Load Balancing Strategy
L4: Supavisor (PG)
Cloud-native PG pooler. Handles millions of end-client connections. Tenant-aware (user_id routing). Automatic pool failover on node death. GET → replica, writes → primary.
L7: Cloudflare / Nginx
API gateway: rate limiting, auth header validation, geo-routing. Cloudflare Workers for edge auth token validation. Nginx reverse proxy at origin for Supabase REST/Realtime.
LLM Proxy: LiteLLM
Multi-provider routing (Gemini/Claude/GPT). Automatic failover + retry. Cost-based routing (cheapest first). Rate limit per user tier. Separate node(s) from Supabase VM at 1K+ DAU.
Object Storage: CDN
Model binaries (~400MB SLM) served via CDN (Cloudflare R2 + cache). App bundle includes base model; OTA updates via CDN. LoRA adapters (~5-20MB) via presigned R2 URLs.
🔄 Scaling Decision Flow
1. PG CPU >70% sustained? → Scale up VM (4 → 8 → 16 vCPU). Cheapest move.
2. PG connections >500 concurrent? → Add Supavisor pooler (if not already present).
3. Read latency >100ms p95? → Add read replica + route GETs to replica.
4. LLM API latency spikes? → Add LiteLLM failover node + multi-provider routing.
5. Multi-region users? → Add geo-distributed replica (Mumbai + Singapore). Supavisor geo-routes.
6. >100K DAU? → Evaluate managed PG (AWS Aurora / Supabase hosted) + tenant sharding.
