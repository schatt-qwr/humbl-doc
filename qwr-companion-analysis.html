1 — What QWR Companion Does Today
Overview

QWR Companion is a production companion app for smart glasses (DPVR G1 + Mentra Live). It is Android-primary with iOS support. Package name: g1_glasses_app. It has real Supabase auth, real BLE device connections, real voice processing, and ships to users.

Voice Intent Pipeline
Activation → Audio Capture → STT (Whisper) → Intent → Execute
Activation: Button press, wake word (Porcupine/Picovoice, placeholder), continuous listening (VOSK)
Audio: Phone mic or glasses mic (BLE audio stream)
STT: whisper_flutter_new (on-device Whisper.cpp, tiny/base/small models)
Intent: Offline regex engine (IntentRegistry, ~30 intents) + online AI (Gemini/OpenAI)
Execution: CommandExecutor → CommandExecutorDelegate (AppStateProvider)
TTS: flutter_tts (system TTS) for spoken responses

1B — Connectors & Tools: Same Thing, Different Wrapper
Key Insight: Connectors ARE Tools

QWR’s “connectors” and humbl_core’s “tools” do the same thing — execute actions against external services or hardware. The only difference is the wrapper around them:

QWR ConnectorsCore Tools
DiscoveryHardcoded in CommandExecutor switch/caseLLM-discoverable via MCP schema + JSON Schema
SecurityNone — any code can call any connector3-gate (access level + permission state + resource lease)
Invocationexecutor.execute(intent) directlyPipeline → ExecuteToolNode → ToolRegistry
SchemaNo formal schema, ad-hoc parametersJSON inputSchema / outputSchema per tool
Tier GatingNo tier controlFree / Standard / Plus / Ultimate per tool
StateisEnabled / isAvailable / hasPermissionToolState (ready / initializing / notSupported / permissionNotGranted)

Migration approach: Connector implementation code moves into humbl_core as proper platform interfaces + tool implementations. QWR companion then consumes them from humbl_core like any other tool. The connectors don’t disappear — they get promoted to first-class tools with MCP discoverability, security gates, and tier control.

Platform / Device Connectors (Native OS APIs)

These interact with the operating system or device hardware via MethodChannel, native SDKs, or Flutter plugins. They belong in humbl_core’s platform/ layer as proper I*Manager interfaces with native implementations.

Migration Path Summary
Platform connectors → Move implementation into humbl_core under platform/ with proper I*Manager interfaces + native Kotlin/Swift. Tools in tools/domains/ consume them.
3rd party connectors → Move into humbl_core as tools/connectors/ with their own lifecycle, auth, and API key management. Register as tools with cloud dependency.
Hybrid connectors → Implement core’s IMediaManager interface, routing internally to the best available music source.
QWR companion → Initializes everything at startup, provides API keys and user context, consumes tools from humbl_core. No longer owns connector logic directly.

Key Architectural Tensions
1. Intent System: QWR uses phrase/regex matching (~30 intents). Core uses MCP tools with structured JSON Schema (81 tools). These CANNOT coexist — core’s system is strictly superior. QWR’s IntentRegistry must be replaced by humbl_core ToolRegistry.
2. Pipeline Model: QWR is a linear pipe (audio → text → intent → action). Core is a graph with loops, conditional routing, multi-step execution, and cloud escalation. QWR’s VoiceIntentPipeline becomes a frontend to core’s StateGraph.
3. State Management: QWR uses mutable ChangeNotifier. Core uses immutable copyWith. QWR can keep Provider for UI state, but pipeline state must flow through PipelineState.
4. Device Abstraction: Both have glasses abstraction layers. Core’s is more general (IPeripheralProvider + capabilities). QWR’s has real BLE working code. Need to merge: QWR adapters implement core’s interfaces.
5. Cloud AI: QWR has working cloud AI (Gemini, OpenAI, WebRTC). Core has none. QWR’s services become the ICloudGateway implementation.

├── Output         TTS + Earcons + Glasses Display + App UI

├── Glasses HW     BLE adapters implementing core's IPeripheralProvider

       │
       │  depends on (pubspec dependency)

├── Security       3-gate access control, plugin verification
├── Platform       13+ manager interfaces (WiFi, BT, Camera, Contacts, Phone, etc.)

QWR Responsibilities (Brain + UI)
1. STT: Whisper on-device speech-to-text (model loading, audio capture, transcription)
2. TTS: System TTS output + earcon sound effects
3. LM Engine: On-device SLM (Qwen3-0.6B via llama.cpp FFI in background isolate)
4. Intent Classification: LmIntentProcessor adapter for pipeline’s ClassifyNode
5. Glasses Hardware: BLE adapters implementing core’s IPeripheralProvider
6. Auth & User: Supabase auth, user tier, BYOK API keys, secure storage
7. UI: All 35+ screens, theming, navigation, onboarding, settings
8. Startup Wiring: Create platform managers, inject into core, start pipeline
Core Responsibilities (Engine + Tools + Connectors)
1. Pipeline: StateGraph orchestration, routing decisions, multi-step loops
2. Tools: 81+ MCP tool definitions, schemas, validation, execution lifecycle
3. Connectors: Contacts, Phone, Notifications, Music (platform + 3rd party) as proper tools
4. Cloud Gateway: ICloudGateway + Gemini/OpenAI implementations for cloud escalation
5. Security: 3-gate access control, plugin verification, access capping
6. Platform Interfaces: 13+ I-prefix abstractions (WiFi, BT, Camera, Contacts, Phone, etc.)
7. Resources: Hardware leasing, contention resolution, lease timeouts
8. Device SDK: IPeripheralProvider contract + capability matching
9. Logging: Structured journal, trace IDs, confidential encryption
10. Models: Model config registry, routing policy

User speaks → QWR VoiceIntentPipeline captures audio
 → QWR WhisperSttEngine transcribes to text
 → QWR creates PipelineState with transcript
 → Core PipelineOrchestrator.runStream()
  → Core ContextAssemblyNode (memory injection)
  → Core ClassifyNode → QWR LmIntentProcessor (on-device SLM)
  → Core RouteDecisionNode (fast / slm_loop / cloud_loop)
  → fast: Core ExecuteToolNode → Core ToolRegistry.execute()

  → cloud: Core CloudGatewayNode → Core ICloudGateway (Gemini/OpenAI impl)
